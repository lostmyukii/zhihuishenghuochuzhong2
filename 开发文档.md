## 1. 先读结论：本项目采用哪条固件路线

本项目正式采用：

```text
N16R8 PIO/Arduino比赛固件
  -> CH340 USB串口单行JSON
  -> Python网关或HTTPS Web Serial
  -> Dashboard

Dashboard麦克风或文本
  -> 网页语音服务STT与自然语义解析
  -> 双重白名单校验
  -> 标准command返回N16R8
```

这条路线中的职责分工是：

- N16R8 PIO固件负责真实传感器采集、情境证据计算、安全中断、执行器控制和串口协议。
- Python网关或浏览器Web Serial负责把串口JSON转发到网页、WebSocket和MQTT。
- Dashboard负责麦克风/文本入口、状态展示和第二次白名单校验；服务器语音接口负责STT和自然语义解析。
- 用户可以使用多种自然说法，但网页链路最终只能输出本项目定义的安全意图和参数，不能直接生成GPIO命令。
- PIO固件本身不宣称运行板内小智语音系统，也不在 `hello` 中宣称板内MCP可用。

### 1.1 与小智统一固件的关系

第一次执行完整PIO上传会写入PIO的bootloader、分区表和Arduino应用，因此会替换开发板当前的小智统一固件运行环境。

必须明确：

- PIO应用地址是 `0x10000`。
- 当前相邻“初中”项目的小智统一固件app地址是 `0x400000`。
- `0x100000`不是当前已经核实的PIO或小智app地址，禁止照抄技能旧稿。
- PIO的 `firmware.bin` 不能写到 `0x400000`。
- 小智的 `xiaozhi.bin` 不能写到 `0x10000`。
- PIO上传即使没有主动擦除旧NVS，也不能承诺板内小智功能继续可用，因为分区表和运行应用已经改变。

首次切换到PIO前，必须先把整片16MB Flash备份到Git仓库外。备份可能包含Wi-Fi和小智激活信息，必须当作敏感文件保存。

## 2. 开发目标与非目标

### 2.1 本阶段目标

1. 建立可重复编译的PlatformIO工程。
2. 固定N16R8板卡、GPIO、依赖、串口和分区基线。
3. 让开发板输出真实 `hello`、`telemetry`、`event` 和 `ack` 单行JSON。
4. 实现六种情境模板、证据覆盖率、情境匹配度和用户纠正。
5. 实现MQ2、水滴、火焰独立安全中断和安全优先级。
6. 实现8键AD、风扇、舵机、继电器、RGB和蜂鸣器真实控制。
7. 提供无板mock、真板网关、Dashboard和网页端小智自然语言链路。
8. 形成“测试通过后再烧录、烧录后必须校验”的工程流程。

- 不把小智ESP-IDF统一固件迁移到PlatformIO。
- 不让PIO固件同时承担板内小智语音对话。
- 不接RFID。
- 不控制 `220V` 负载。
- 不加入摄像头、人脸识别、人数统计或原始声音录制。
- 不在没有计量模块时宣称精确节电量。
- 不让自由文本模型直接控制危险执行器。
- 不把相邻“初中”项目的生产域名、端口、Topic或项目画像直接复用到本项目。

## 4. 固定硬件与GPIO合同

### 4.1 传感器和交互输入

| 模块 | 拓展板端口 | 接线 | 固件符号 | 说明 |
| --- | --- | --- | --- | --- |
| 光敏传感器 | `GPIO1` | `V-G-S` | `PIN_LIGHT` | ADC，明暗和学习照度证据 |
| 声音传感器 | `GPIO4` | `V-G-S` | `PIN_SOUND` | ADC，只采强度，不录音 |
| 温湿度DHT | `GPIO14` | `V-G-S` | `PIN_DHT` | DHT11首阶段基线 |
| PIR人体红外 | `GPIO5` | `V-G-S` | `PIN_PIR` | 数字输入，通道活动证据 |
| 8键AD键盘 | `GPIO10` | `V-G-S10` | `PIN_KEYPAD_ADC` | ADC，情境和参数操作 |
| MQ2烟雾/燃气 | `GPIO2` | `AO -> S2`，`G -> G` | `PIN_MQ2` | ADC，必须先分压到 `0~3.3V` |
| 水滴传感器 | `GPIO8` | `V-G-S` | `PIN_WATER` | 数字输入，触发电平须实测 |
| 火焰传感器 | `GPIO45` | `V-G-S` | `PIN_FLAME` | 数字输入，触发电平须实测 |

### 4.2 执行器

| 模块 | 拓展板端口 | 固件符号 | 说明 |
| --- | --- | --- | --- |
| 有源蜂鸣器 | `GPIO13` | `PIN_BUZZER` | 普通提醒和安全报警；当前实物端口与模块均为`G-V-S`同序 |
| 风扇 | `GPIO11` | `PIN_FAN` | PWM或明确高低电平输出 |
| 舵机 | `GPIO9` | `PIN_SERVO` | PWM，窗帘/通风窗模型 |
| 继电器 | `GPIO12` | `PIN_RELAY` | 只控制低压模型负载 |
| RGB灯环 | `GPIO46` | `PIN_RGB` | WS2812/NeoPixel类灯环 |

### 4.3 RFID与可选OLED

- 当前不接RFID，`GPIO11/12/13`固定给风扇、继电器和蜂鸣器。
- OLED不在已确认的首阶段硬件合同中，不作为PIO首阶段必需依赖。
- 将来确认使用板载OLED时，再单独加入I2C引脚、库、显示协议和验收项，不能先在文档里宣称已实现。
- 当前GPIO13接线必须按实物丝印逐针核对为`G→G、V→V、S13→S`；通用的`V/G/S`描述表示三类端子，不代表所有模块的物理排列顺序。
- 当前RGB灯环的两个信号端是字母`SI`（输入）和`SO`（输出），不是数字`S1/S0`。主控信号必须接`SI`，`SO`只用于向下一个灯环级联。

### 4.4 MQ2分压

MQ2使用5V供电时，AO禁止直接进入GPIO2。建议使用留有误差余量的分压网络，例如上臂 `12k`、下臂 `20k`：

```text
MQ2 AO ---- 12k ----+---- GPIO2 / S2
                    |
                   20k
                    |
GND ----------------+---- GND
```

5V输入时理论节点电压约为3.125V。最终必须用万用表实测，并确认最坏情况下不超过3.3V。

## 11. 固件模块职责

### 11.1 `main.cpp`

只负责：

- 初始化串口和模块。
- 调用各模块 `begin()`。
- 按非阻塞节奏调度采样、判断、执行和遥测。
- 不在主循环中使用长时间 `delay()`。

主循环结构：

```cpp
void loop() {
  protocol.pollSerial();
  sensors.poll();
  keypad.poll();
  contextEngine.update();
  safetyEngine.update();
  actuators.apply();
  protocol.emitDueFrames();
}
```

### 11.2 `sensors.cpp`

- 读取光敏、声音、PIR、8键AD、MQ2、水滴和火焰。
- DHT独立低频读取。
- 保存原始值、归一化值、有效标志和数据年龄。
- 维护触发去抖、迟滞和连续帧计数。
- 不在采样层直接控制执行器。

### 11.3 `context_engine.cpp`

- 读取有效传感器快照。
- 计算各情境覆盖率和匹配度。
- 输出支持、反向和缺失证据代码。
- 处理 `possible`、`ambiguous`、`evidence_missing`、`confirmed`、`corrected`、`unknown`。
- 不直接写GPIO。

### 11.4 `safety_engine.cpp`

- 维护MQ2预热、燃气/烟雾、水滴、火焰和数据中断状态。
- 处理连续确认和持续恢复。
- 输出最高优先级和全部有效风险。
- 阻止普通模式与手动命令覆盖安全动作。

### 11.5 `actuators.cpp`

- 集中控制风扇、舵机、继电器、RGB和蜂鸣器。
- 每个输出都区分自动目标、手动目标和安全覆盖目标。
- 上电先写安全状态，再允许其他模块工作。
- 遥测必须返回最终实际目标，而不是只返回用户请求。

### 11.6 `keypad.cpp`

- 把GPIO10 ADC值映射为A/B/C/D/上/下/左/右。
- 使用稳定窗口和防重复触发。
- 输出按键事件，不直接操作全局GPIO。

### 11.7 `protocol.cpp`

- 输出 `hello`、`telemetry`、`event`、`ack`。
- 解析单行JSON `command`。
- 保留命令ID并返回同ID的 `ack`。
- 协议行不混入调试文字。

## 12. 核心数据类型

建议在 `project_types.h` 中集中定义：

```cpp

  uint8_t coverage;
  uint8_t match;
};
```

支持证据、反向证据和缺失证据使用固定枚举或固定字符串代码，不能在固件中生成任意自然语言。中文解释由Dashboard纯函数模块完成。

## 13. 采样与数据有效性

### 13.1 快速传感器

光敏、声音、PIR、8键AD、MQ2、水滴和火焰建议每200ms采样。遥测仍约每500ms输出一次，避免串口被高频JSON淹没。

### 13.2 DHT

- 每2000ms最多读取一次。
- 短暂失败时保留最近有效值和数据年龄。
- 启动后从未成功，或最近成功时间超过6000ms，设置 `dhtValid=false`。
- DHT无效时不参与湿热通风评分。

### 13.3 数字触发输入

水滴和火焰模块的有效电平必须在实物上标定。相邻原项目当前模块为：

- 水滴：空闲低，触发高。
- 火焰：空闲低，触发高。

本项目可以把高电平作为初始基线，但第一次真板接线后必须验证，并把 `waterInputLevel`、`waterTriggerLevel`、`flameInputLevel`、`flameTriggerLevel` 输出到 `telemetry.health`。

### 13.4 模拟量校准

光敏、声音、MQ2和8键AD不得直接假设 `analogRead()` 全量程都有效。开发时记录：

- 空闲原始值。
- 受控触发原始值。
- 当前归一化区间。
- 电源电压和模块电位器位置。
- 现场阈值和迟滞范围。

## 14. 情境证据计算

每个模板包含：

- 必需证据。
- 可选证据。
- 满足条件。
- 权重。
- 最低覆盖率。
- 确认阈值。
- 模糊区间。

计算公式：

```text
coverage = 当前有效证据权重 / 模板全部证据权重 × 100%
match    = 已满足证据权重 / 当前有效证据权重 × 100%
```

判断顺序：

1. 必需传感器是否有效。
2. 覆盖率是否达到模板要求。
3. 各模板匹配度排序。
4. 最高分是否达到确认阈值。
5. 前两名差距是否小于模糊区间。
6. 是否有用户确认或纠正。

用户纠正只写入事件和配置建议。只有用户明确保存且收到板端 `ack` 后，阈值或权重才生效。

## 15. 六种情境的工程映射

| 情境 | 固件名 | 主要证据 | 普通动作 |
| --- | --- | --- | --- |
| 实时侦测 | `detect` | 全部有效证据 | 只比较情境，不执行普通强动作 |
| 专注学习 | `study` | PIR、光照、声音，DHT辅助 | 低压学习灯、RGB橙色提醒、允许的通风 |
| 安静休息 | `rest` | 低声音、低照度、较少活动 | RGB低亮、窗帘休息位、普通蜂鸣静音 |
| 湿热通风 | `ventilation` | DHT、PIR、用户阈值 | 风扇分级、舵机开窗、RGB青色 |
| 无人节能 | `energy` | PIR持续无活动、光照、执行器状态 | 关闭低压负载、风扇和非必要灯光 |
| 自定义情境 | `custom` | 用户选择至少两项证据 | 白名单内的低压动作 |

普通情境只能提出普通动作，最终输出必须经过安全引擎仲裁。

### 15.1 阶段3传感器与情境基线

阶段3把固件拆为以下可独立验证单元：

- `sensors.cpp`：200ms快速采样GPIO1/4/5/10/2/8/45，DHT11每2000ms单独读取；短暂失败保留最近有效值，超过6000ms退出有效状态。
- `input_filter.h`：水滴与火焰连续3帧确认、连续3帧恢复；高电平仅是相邻实物经验形成的初始基线，遥测始终标记 `high-unverified`。
- `context_engine.cpp`：纯C++情境计算，不依赖Arduino或GPIO，输出 `possible/ambiguous/evidence_missing/unknown`、覆盖率、匹配度和固定证据代码。
- 阶段3当时的 `main.cpp` 只组织串口协议、采样和情境计算，不写任何执行器GPIO。

阶段3暂定原始值阈值只用于让算法可以编译和行为测试，不能当作实物标定结论。`telemetry.health`固定输出：

```json
{"stage":"stage3-sensors-context","sensorSampling":"real-gpio-unverified","thresholdProfile":"provisional-unverified","hardwareVerified":false,"calibrationRequired":true,"actuatorsReady":false,"safetyReady":false}
```

MQ2启动30000ms内输出 `mq2State=warming` 和剩余时间，不生成燃气报警；水滴、火焰输出去抖后的传感值及当前输入电平。

### 15.2 阶段4软件基线

阶段4新增三个职责分离的单元：

- `SafetyEngine`：纯C++安全状态机，处理MQ2预热、`2600/2400`迟滞、3个新采样确认/恢复、1500ms安全输入失效以及火焰 > MQ2 > 水滴 > 传感器故障优先级。
- `ActuatorPlanner`：先生成普通情境目标，再按字段应用安全覆盖；静音只把蜂鸣目标改为关闭，不删除风险原因、继电器保护或RGB目标。
- `ActuatorDriver`：`0.3.0`软件基线默认未武装，只返回`unarmed`，当时源码中不包含GPIO、PWM、舵机attach或NeoPixel写入。

`0.3.0`软件基线曾冻结总武装开关和五个独立开关为`false`。该阶段的状态含义为：

- `actuatorTargets`表示安全仲裁后的逻辑计划。
- `actuators`表示物理实际值；当前五项都为 `null`，不得由目标值推断实际输出。
- `safetyReady=true`仅表示安全软件通过测试。
- `actuatorsArmed=false`、`actuatorsReady=false`、`hardwareVerified=false`、`calibrationRequired=true`保持真实。
- `set.buzzerEnabled`可在未武装时更新安全静音配置；合法物理执行器命令返回 `actuators_unarmed`，非法形状或值返回 `invalid_actuator_command`。

### 15.3 阶段4 GPIO13真板验收

固件`0.3.1`只新增GPIO13物理边界，保持其余执行器未武装：

- `ACTUATORS_ARMED=true`、`BUZZER_ARMED=true`、`BUZZER_HARDWARE_VERIFIED=true`。
- `FAN_ARMED/SERVO_ARMED/RELAY_ARMED/RGB_ARMED=false`；驱动源码不得写GPIO11/9/12/46。
- 初始化先`digitalWrite(PIN_BUZZER, LOW)`，再`pinMode(PIN_BUZZER, OUTPUT)`，避免输出模式切换时误鸣。
- `actuator.buzzer=true`触发800ms非阻塞高电平窗口；同ID成功`ack`包含`buzzerPulseMs=800`，到期立即输出`buzzerOn=false`遥测。
- `actuator.buzzer=false`立即停止，但不改变`buzzerEnabled`安全静音配置。
- 安全规划中的`alarm/intermittent`本轮仍只保留在`actuatorTargets`，不会自动驱动蜂鸣器。
- `hello.features.physicalActuators=false`继续表示整组执行器未完成，另以`physicalBuzzer=true`精确声明GPIO13可用。
- `actuators.buzzerOn`为真实布尔值；其余四项实际值保持`null`。整组`actuatorsReady=false`、全局`hardwareVerified=false`、`calibrationRequired=true`继续保持真实。

2026-07-15真板证据：候选应用和最终应用均只写`0x10000`应用区并独立校验；上电人工确认静音。首轮150ms命令的回执和自动关闭正常但未听到声音，因此按相邻已验证工程的实物时长调整到800ms。复测收到同ID成功回执，人工确认听到一次蜂鸣并自动停止；显式停止成功，风扇35%命令被`actuators_unarmed`拒绝。

### 15.4 阶段4 RGB灯环诊断结果

2026-07-15在用户明确授权下完成RGB低压诊断。最初把灯环丝印字母`SI/SO`误读为数字`S1/S0`，信号接在`SO`输出端时，GPIO46候选固件虽能返回同ID成功回执、红色状态和约5秒自动关闭状态，但灯环没有可见输出。

纠正为`SI`输入后，以临时候选`0.3.2-rc4-pin13-diagnostic`复用已验收GPIO13端口；同一灯环、同一线材在GPIO13出现人工可见红灯，证明灯环、供电、线材、`SI`方向和NeoPixel基本驱动链路可工作。随后以`0.3.2-rc5-gpio46-si`接回正式GPIO46，同样取得输出引脚46、成功回执、红色状态、约4.98秒自动关闭和显式关闭等机器证据，但用户未看到红灯。

因此GPIO46的RGB实物验收未通过，不能把软件状态当作物理亮灯证据。安全正式版`0.3.2`已恢复`RGB_ARMED=false`、`RGB_HARDWARE_VERIFIED=false`；真板确认`rgbState=null`，RGB命令返回`actuators_unarmed`，蜂鸣器保持关闭。下一步应在新授权下用已知良好低压负载检查GPIO46基础输出/拓展板链路，或正式评审RGB改用其他GPIO；在此之前不进入RGB自动联动。

## 16. 安全状态机与优先级

本节动作表在阶段4表示 `actuatorTargets` 逻辑目标，不代表GPIO已经输出。只有后续逐项实物验收并武装对应驱动后，才能把执行结果写入 `actuators`。

### 16.1 上电默认值

- 风扇关闭。
- 继电器断开。
- 蜂鸣器物理静音，但 `buzzerEnabled=true`。
- RGB关闭或安全低亮。
- 舵机先不驱动，确认供电和安全角度后再attach。

### 16.2 风险联动

| 风险 | 风扇 | 舵机 | 继电器 | RGB | 蜂鸣器 |
| --- | --- | --- | --- | --- | --- |
| MQ2预热 | 关闭 | 不动作 | 保持安全默认 | 黄色 | 静音 |
| MQ2确认 | 100%排风 | 打开通风窗 | 断开 | 红色 | 报警，除非明确静音 |
| 火焰确认 | 强制停止 | 关闭 | 断开 | 红色 | 报警，除非明确静音 |
| 水滴确认 | 不执行无关动作 | 不执行无关动作 | 断开 | 蓝红 | 间歇报警，除非明确静音 |
| 数据中断 | 停止普通自动动作 | 安全位置/不驱动 | 断开 | 灰色/关闭 | 不冒充风险结论 |

### 16.3 冲突规则

1. 火焰优先于MQ2排风，火焰确认后风扇停止。
2. 多个风险可同时显示，不能只保留最高风险代码。
3. 风险恢复需要持续安全状态，不能以单帧恢复或拔掉传感器解除。
4. `set.buzzerEnabled=false`是明确安全静音。
5. `actuator.buzzer=false`只停止手动/测试声音，不改变安全静音配置。
6. 手动风扇、舵机、继电器和RGB命令不能覆盖安全目标。

## 17. 执行器实现要求

### 17.1 风扇

- 将有效电平做成显式配置，兼容高电平或低电平有效模块。
- PWM目标使用 `0~100` 百分比，底层统一换算。
- 安全排风强制100%，火焰确认强制0%。
- 遥测返回最终实际目标和有效电平配置。

### 17.2 舵机

- 初始不attach，避免上电抖动。
- 首次attach前确认独立供电和共地。
- 角度范围在实物模型上标定，不直接假设0/180都安全。
- 区分窗帘休息位、通风位、安全关闭位。

### 17.3 继电器

- 只接低压模型负载。
- 安全状态下保持断开。
- 手动命令必须经过安全引擎。
- 遥测说明当前是自动、手动还是安全覆盖。

### 17.4 RGB

- 限制亮度，避免USB供电压力。
- 安全红色优先于模式色和手动色。
- 减少动画阻塞，不使用长时间同步延迟。

### 17.5 蜂鸣器

- 上电物理静音。
- 普通提醒和安全报警分别处理。
- `buzzerEnabled`默认true。
- 页面静音后仍保留风险卡、继电器保护和RGB安全状态。
- 当前有源模块按GPIO13高电平有效验收，800ms短测使用`millis()`计时，不用阻塞式`delay()`。
- 当前只验收手动测试窗口；自动安全报警波形和间歇节奏需在其他执行器安全覆盖一并验收后再启用。

允许的最终意图：

- `queryContext`
- `explainContext`
- `setMode`
- `confirmContext`
- `correctContext`
- `setThreshold`
- `querySafety`
- `muteBuzzer`
- `unknown`

## 34. USB和串口故障恢复

### 34.1 `Device not configured`或端口消失

1. 停止当前命令。
2. 断开执行器负载。
3. 更换短数据线或USB接口。
4. 重新运行端口查找命令。
5. 注意端口可能从 `usbserial-110` 变成 `usbserial-130`。
6. 用低速无stub `chip_id`做只读检查。

```bash
"$PIO_PY" "$ESPTOOL" \
  --chip esp32s3 -p "$PORT" -b 115200 \
  --before default_reset --after hard_reset --no-stub chip_id
```

没有串口设备意味着优先处理物理连接或供电，而不是修改固件。

### 34.2 端口被占用

- 关闭PIO Monitor。
- 关闭Python网关。
- 检查 `lsof`。
- 不同时打开 `/dev/cu.*`和对应 `/dev/tty.*`。

### 34.3 烧录成功但无JSON

依次检查：

- 是否保留CH340构建标志。
- Monitor波特率是否115200。
- 是否打开了错误端口。
- 正式串口是否混入大量调试文本。
- `setup()`是否先 `Serial.begin(115200)`再输出 `hello`。
- 网关是否只接受了其他项目的 `project`。

### 34.4 语音能识别但设备不动作

PIO路线不检查板内MCP，依次检查：

- `/api/voice/transcribe`是否返回了正确文字。
- `/api/voice/intent`是否生成了白名单意图，是否发生规则降级或返回 `unknown`。
- Dashboard第二次白名单校验是否通过。
- 网关是否生成带ID的 `command`。
- 串口是否写入成功。
- 固件是否返回 `ack`。
- `telemetry.actuators`是否变化。
- 安全引擎是否合理阻止了当前动作。

## 41. 阶段5一体化实时系统开发合同

### 41.1 实时链路

```text
N16R8 / CH340
  -> Chrome/Edge Web Serial -> Dashboard
  -> Python串口网关（备用）-> 本地WebSocket
  -> 后续独立WSS/MQTT -> 第二浏览器/移动端
```

主路是网页点击“连接开发板”后的 Web Serial。网页和 Python 网关复用相同的单行 JSON 协议，不分叉定义两套命令。

### 41.2 风扇硬件口径

```text
GPIO11 -> 风扇PWM
外部电池VM -> 风扇VM
外部电池GND -> 风扇GND -> 主板GND（共地）
主板VM不接风扇VM
```

外接电池电压必须符合风扇模块额定要求。开发、编译和烧录期间默认关闭外接电池。

### 41.3 执行器启动策略

- 完整固件同时包含风扇GPIO11、舵机GPIO9、继电器GPIO12、蜂鸣器GPIO13和RGB GPIO46驱动。
- 上电先写安全值，再设置输出模式；舵机在安全窗内不得突然移动。
- `ACTUATOR_BOOT_GUARD_MS=5000`，保护窗期间 `actuatorsReady=false`。
- 保护窗结束后一次性启用普通情境动作和安全覆盖。
- 手动调试使用白名单，`auto`只清除对应执行器的手动覆盖；安全覆盖永远优先。
- `rgbHardwareVerified=false`继续保持真实，命令和内部状态不等于肉眼可见的灯环证据。

### 41.4 Dashboard功能

- 顶部开发板/Web Serial/WebSocket/MQTT/数据流状态。
- 情境侦探房屋热区和无摄像头说明。
- 全部传感器的值、GPIO、有效性、数据年龄和情境参与状态。
- 覆盖率、匹配度、支持/反对/缺失证据和安全原因。
- 六种情境、执行器调试台、设备注册、语音文本测试、当前状态和事件日志。
- 页面在`3500ms`无新鲜遥测后清除旧结论，不把WebSocket或USB标签当作开发板在线。
