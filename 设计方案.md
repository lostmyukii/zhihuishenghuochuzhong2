# 智慧生活初中组：N16R8 无摄像头家庭情境侦探屋设计方案

> 方案状态：已完成方向与四部分设计确认，当前作为“初中2组”独立作品的正式设计基线。  
> 作品副标题：用环境线索理解生活状态的隐私友好型智能系统。  
> 作品载体：N16R8 / LY-AS-ESP32-S3 v1.2 与家庭房屋沙盘。  
> 硬件口径：传感器接拓展板上印有对应 GPIO 编号的 `V/G/S` 三针口；本项目不接 RFID。  
> 设计依据：智慧生活初中组的多维度数据采集、多模式控制、数据可视化、场景联动控制和自定义适配五项能力。  
> 当前边界：本文件定义作品命题、情境模型、硬件角色、数据链路、展示结构和验收标准；不表示固件、Dashboard 或真板烧录已经完成。

## 1. 作品名称与核心命题

作品名称：

**N16R8 无摄像头家庭情境侦探屋——用环境线索理解生活状态的隐私友好型智能系统**

普通智能家居常见两类问题：

1. 依赖摄像头观察家庭空间，容易带来隐私顾虑。
2. 使用“一个传感器对应一个动作”的简单规则，容易忽略其他环境线索并产生机械联动或误判。

本作品不采集人脸、不通过声音传感器录音，而是组合光照、声音强度、温湿度、人体活动以及安全传感器，形成一条可以被用户看懂的“环境证据链”。系统不只给出结果，还要展示判断依据、反向证据和缺失证据，并允许用户确认、纠正和保存自己的生活规则。

核心流程为：

```text
采集多种非图像环境线索
  -> 检查数据是否有效和新鲜
  -> 计算各情境的证据覆盖率与匹配度
  -> 展示支持证据、反向证据和缺失证据
  -> 执行当前情境允许的低压动作
  -> 用户确认或纠正判断
  -> 保存经过确认的个性化配置
```

作品的教育价值是让学生理解：智能系统不能只看一个数值就替人下结论，必须说明“看到了什么、为什么这样判断、还有什么证据不足、用户如何纠正”。

## 4. 固定硬件与接线口径

### 4.1 传感器与交互输入

| 模块 | 拓展板端口 | 接线 | 本项目角色 |
| --- | --- | --- | --- |
| 光敏传感器 | `GPIO1` 端口 | `V-G-S` | 判断明暗、学习照度和夜间条件 |
| 声音传感器 | `GPIO4` 端口 | `V-G-S` | 只采集声音强度或变化，不录音、不识别谈话内容 |
| 温湿度 DHT | `GPIO14` 端口 | `V-G-S` | 判断温湿度背景和通风需求 |
| PIR 人体红外 | `GPIO5` 端口 | `V-G-S` | 判断指定通道是否出现人体活动，不识别人、不统计人数 |
| 8键 AD 键盘 | `GPIO10` 端口 | `V-G-S10` | 选择情境、切换证据焦点、调整参数和纠正判断 |
| MQ2 烟雾/燃气 | `GPIO2` 端口 | `AO -> S2`，`G -> G`，供电按模块要求 | 安全中断条件；预热完成前只显示数据，不进入危险判断 |
| 水滴传感器 | `GPIO8` 端口 | `V-G-S` | 漏水安全中断条件 |
| 火焰传感器 | `GPIO45` 端口 | `V-G-S` | 受控演示中的火焰/红外触发信号，不判断真实火势等级 |

### 4.2 执行器

| 模块 | 拓展板端口 | 本项目角色 |
| --- | --- | --- |
| 有源蜂鸣器 | `GPIO13` | 分级提醒与安全报警 |
| 风扇 | `GPIO11` | 湿热通风或MQ2安全排风 |
| 舵机 | `GPIO9` | 模拟窗帘或通风窗 |
| 继电器 | `GPIO12` | 只切换低压模型负载，如学习灯或保护指示负载 |
| RGB灯环 | `GPIO46` | 显示情境、关注、异常、安全中断和夜间低扰动状态 |

### 4.3 不使用 RFID

当前项目不接 RFID，也不占用 `GPIO11/12/13` 做 RFID SPI。三个端口固定用于风扇、继电器和有源蜂鸣器。

### 4.4 MQ2 分压要求

如果 MQ2 使用 `5V` 供电，`AO` 不得直接进入 `GPIO2`。必须使用经过计算并实测确认的分压网络，把最大输入限制到 `0~3.3V`。为给电源波动和电阻误差留出余量，设计阶段建议使用上臂 `12k`、下臂 `20k` 或具有同等安全余量的组合：

```text
MQ2 AO ---- 12k ----+---- GPIO2 / S2
                    |
                   20k
                    |
GND ----------------+---- GND
```

上述示例在输入为 `5V` 时，分压节点的理论电压约为 `3.125V`。如果沿用原项目常见的上臂 `10k`、下臂 `20k` 组合，其理论值约为 `3.33V`，必须结合实际模块输出、电源偏差和电阻误差进行测量；无法证明最坏情况下不超过 `3.3V` 时，应改用留有余量的阻值组合。

## 5. 必须遵守的安全边界

1. 继电器只控制低压灯、低压模型负载或状态指示，不连接 `220V`。
2. 风扇、舵机、继电器或RGB使用外部电源时，外部电源必须与 N16R8 共地。
3. 接线前断电，确认 `V/G/S` 线序和供电电压后再连接 USB-C。
4. 上电默认状态为风扇关闭、继电器断开、蜂鸣器物理静音、RGB安全状态、舵机处于经过实物确认的安全角度。
5. “蜂鸣器上电静音”不等于禁用安全报警。安全报警默认保持可用，只有明确的 `buzzerEnabled=false` 才表示用户主动静音。
6. 现场不得使用真实燃气、危险烟雾或明火。MQ2和火焰场景使用安全、可控的模块测试；条件不允许时使用 mock 演示软件闭环。
7. 水滴传感器可使用少量清水安全触发，但必须避免水接触主板、拓展板和裸露供电端。
8. 本作品不代替专业消防、燃气、漏水、安防或人员看护设备。

## 6. 系统状态模型

系统不使用一个 `mode` 变量包办全部含义，而是明确区分四类状态：

```text
selectedTemplate  用户主动选择的目标情境
candidateContext  系统根据有效证据计算出的可能情境
confirmedContext  用户确认或纠正后的结果
safetyState       独立于普通情境的安全中断状态
```

例如用户选择“专注学习”，系统判断的是“当前环境对学习情境的匹配程度”，不能据此断言屋内的人正在学习。

安全中断不属于普通情境评分的一部分。MQ2、水滴或火焰确认后，`safetyState` 可以覆盖普通执行动作，但不能删除原有情境证据和历史记录。

## 7. 情境证据计算

### 7.1 两个独立指标

为防止传感器缺失时匹配度被错误放大，每个情境同时计算两个指标：

```text
证据覆盖率 coverage
= 当前有效证据权重 / 该情境全部证据权重 × 100%

情境匹配度 match
= 已满足证据权重 / 当前有效证据权重 × 100%
```

只有覆盖率达到该模板的最低要求，且必需传感器有效，系统才允许输出“可能情境”。最低覆盖率、确认阈值和模糊区间应在实物标定后确定，不在设计阶段编造永久固定数值。

### 7.2 结果状态

| 状态 | 含义 | Dashboard表达 |
| --- | --- | --- |
| `possible` | 覆盖率足够，某情境匹配度明显领先 | “当前更接近：专注学习” |
| `ambiguous` | 前两项结果接近，证据不能明确区分 | “存在多种可能，请确认” |
| `evidence_missing` | 必需传感器失效或覆盖率不足 | “证据不足，缺少：……” |
| `confirmed` | 用户确认当前候选结果 | “用户已确认” |
| `corrected` | 用户否定候选并指定正确模板 | “用户已纠正，尚未自动改写规则” |
| `unknown` | 没有新遥测或系统状态不可判断 | “等待实时数据” |

### 7.3 用户纠正原则

- 用户纠正首先形成一条可追溯事件记录。
- 系统可以建议调整阈值或权重，但不能在后台静默改变规则。
- 只有用户明确保存，并且板端返回真实 `ack`，新配置才显示“已生效”。
- 本项目当前使用可解释的规则和证据评分，不把纠正过程包装成未经验证的机器学习训练。

## 8. 六种情境模板

### 8.1 实时侦测

目标：比较全部情境的证据覆盖率和匹配度，同时证明数据来自真实开发板。

- 展示所有有效传感器、最近采样时间和端口健康。
- 比较专注学习、安静休息、湿热通风、无人节能和自定义模板。
- 默认只显示判断和建议，不执行普通强动作。
- 安全中断仍由板端实时工作。
- 没有新 `telemetry` 时显示“等待实时数据”，不使用默认值冒充在线设备。

### 8.2 专注学习

主要证据：PIR近期活动、光照、声音强度，DHT作为辅助环境证据。

默认联动：

1. 指定区域出现活动，证明该区域可能有人使用。
2. 光照不足时，继电器开启低压学习灯模型，或舵机调整窗帘模型。
3. 声音超过学习阈值时，RGB显示橙色提醒；是否发出短促蜂鸣由用户配置决定。
4. 温度较高且DHT数据有效时，可进入低强度通风建议或执行允许的风扇动作。
5. Dashboard解释每个动作来自哪项证据，不能只显示“学习模式已开启”。

### 8.3 安静休息

主要证据：较低声音、较低照度和较少人体活动。

默认联动：

- RGB使用低亮度蓝色或用户配置的低扰动颜色。
- 舵机将窗帘模型移动到休息位置。
- 普通蜂鸣提醒保持静音。
- 风扇按温湿度和用户配置决定低速或停止，不能只因为进入休息模板就忽略明显湿热。
- 安全报警不受普通静音影响。

### 8.4 湿热通风

主要证据：有效温度、有效湿度、PIR活动和用户阈值。

默认联动：

- DHT达到用户设定条件后，风扇分级运行。
- 舵机打开通风窗模型。
- RGB显示青色通风状态。
- 无人时是否继续低速通风由用户配置决定，并在看板上说明原因。
- DHT失效时退出自动通风判断，不沿用无限期的旧数据。

### 8.5 无人节能

主要证据：PIR在可配置时间内无活动、光照和当前执行器状态。

默认联动：

- 关闭低压继电器负载。
- 关闭风扇和不必要的RGB灯光。
- 舵机进入用户配置的节能位置。
- 记录设备关闭时长、运行时长和策略执行次数。
- 没有专业电能计量模块，因此不宣称精确节省多少度电或节省多少费用。

### 8.6 自定义情境

用户至少选择两项有效传感证据，并配置：

- 模板名称。
- 必需证据与可选证据。
- 阈值、权重和持续时间。
- 提醒颜色和提醒方式。
- 允许执行的风扇、舵机、低压继电器、RGB或蜂鸣动作。
- 是否需要用户确认。

自定义模板不能修改固定GPIO、解除安全中断、把失效传感器当作正常，或允许危险状态下强制恢复普通动作。

## 9. 安全中断状态机

| 安全状态 | 确认条件 | 确认后的动作 |
| --- | --- | --- |
| MQ2预热 | 模块刚上电，预热未完成 | RGB黄色、Dashboard显示预热；不启动风扇、继电器或蜂鸣器 |
| MQ2持续超阈 | 多帧持续满足当前阈值和确认时间 | 风扇排风、舵机打开通风窗、继电器切断低压负载、RGB红色、蜂鸣报警 |
| 火焰持续触发 | 多帧确认触发，且输入电平经过实物标定 | 风扇停止、舵机关闭、继电器断开、RGB红色、蜂鸣报警 |
| 水滴持续触发 | 多帧确认水滴输入 | 继电器断开、RGB蓝红提示、蜂鸣器间歇报警；不启动无关风扇或舵机 |
| 数据中断 | 超过规定时间没有新鲜遥测 | 状态未知，停止普通自动动作并显示离线；不能显示“环境正常” |

恢复规则：

- 单帧恢复不立即解除中断，应等待持续安全状态。
- 数据断开或拔掉传感器不能当作恢复正常。
- 火焰与MQ2同时触发时，火焰处置优先，风扇必须停止。
- 水滴与其他风险同时出现时，继电器保持低压保护断开，并同时展示全部风险源。
- 明确静音只停止蜂鸣器声音，不清除风险、事件或继电器保护。
- 普通模式、手动按钮、语音和自定义规则均不能覆盖已确认的安全动作。

## 10. 8键AD交互设计

| 按键 | 功能 |
| --- | --- |
| `A` | 返回实时侦测 |
| `B` | 选择专注学习 |
| `C` | 选择安静休息 |
| `D` | 选择无人节能 |
| 左 | 向前切换证据、阈值或权重焦点 |
| 右 | 向后切换证据、阈值或权重焦点 |
| 上 | 增加当前参数 |
| 下 | 减少当前参数 |

湿热通风与自定义情境可通过Dashboard或小智语音入口选择，避免8个按键承担过多复合操作。

Dashboard必须同时显示：当前按键、参数焦点、旧值、新值、命令编号和 `ack` 状态。网页数值变化不等于开发板配置已经改变。

## 11. 数据采集节奏与有效性

- 光敏、声音、PIR、水滴、火焰、8键AD等快速输入可以在 `200~500ms` 周期内采集。
- DHT必须使用独立的低频读取节奏，建议不少于 `2000ms` 一次，避免过快轮询导致反复失败。
- DHT短暂失败时可以保留最近一次有效值，但必须标注数据年龄；持续失效后退出情境评分。
- MQ2必须维护预热状态和剩余/已完成标识，预热期间不产生危险动作。
- 对易抖动输入使用连续帧确认、迟滞或恢复时间，不以单次采样来回切换情境或风险。
- 串口遥测建议约 `500ms` 输出一帧，但DHT值只在新的有效读取出现时更新。


开发板必须返回带相同命令编号的 `ack`：

```json
{"type":"ack","id":"cmd-002","ok":true,"applied":{"soundThreshold":60}}
```

协议规则：

- 每行只输出一个完整JSON对象，不把调试文字混进协议行。
- Dashboard、网页语音服务、网关和固件使用同一组模式名与参数名。
- `actuator.buzzer=false` 只停止当前手动/测试声音，不能代替 `buzzerEnabled=false`。
- 安全报警和普通情境不共用同一个模糊状态字段。
- 未知报警代码继续显示为“设备上报异常：代码”，不能静默丢弃。

## 13. 数据链路与离线能力

推荐复用经过验证的连接思路，但使用本项目独立画像：

```text
传感器
  -> N16R8本地采集、情境评分和安全判断
  -> 115200串口单行JSON
  -> HTTPS Dashboard Web Serial 或本地Python网关
  -> WebSocket / MQTT
  -> 电脑、平板或手机看板

浏览器麦克风或文本
  -> 网页语音服务完成STT与自然语义解析
  -> 安全白名单voiceIntent
  -> Dashboard / 网关转换为带ID的command
  -> N16R8执行并返回ack + telemetry
```

关键边界：

- MQ2、火焰和水滴安全判断在开发板本地完成，网络中断时板端仍可执行安全动作。
- 使用Chrome或Edge的电脑可通过HTTPS页面授权Web Serial；Safari或手机可作为远程看板，但不作为USB直连网关。
- WebSocket、MQTT和开发板在线状态分别显示。
- 只有新鲜 `hello` 或 `telemetry` 才能证明开发板在线。
- 网页返回的自身广播帧必须去重，避免事件重复计数或动画重复播放。
- 本地USB控制在WSS或MQTT不可用时仍应继续工作。
- 自然语言的理解不在N16R8主板上运行；浏览器发起识别并展示结果，服务器语音接口负责STT和语义解析，浏览器与服务器对输出意图进行双重白名单校验。
- 语音服务不可用时，主板本地采集、安全中断、8键AD、Dashboard按钮和串口控制仍可工作；网页应明确显示“语音服务不可用”，不得伪造识别结果。
- 公网部署时另行分配独立域名、路径、端口和Topic，不直接复用原“初中”项目的生产标识。

## 14. Dashboard信息架构

视觉可继续采用iOS/iPadOS风格的大圆角、毛玻璃和蓝白银基调，但首屏必须是“情境推理控制台”，不是营销首页，也不是原智能住宅设备总览。

### 14.1 首屏结构

```text
顶部状态条
  开发板 / USB / WebSocket / MQTT / 数据新鲜度 / 安全状态

左侧：家庭环境证据地图
  房间位置 / 传感器值 / 有效或失效 / 当前证据角色

右侧：当前情境推理卡
  候选情境 / 覆盖率 / 匹配度
  支持证据 / 反向证据 / 缺失证据
  用户确认 / 用户纠正

中部：动作解释与变化时间轴
  发现证据 -> 形成判断 -> 执行动作 -> 用户确认或纠正

底部
  六种情境入口 / 实时与历史趋势 / 8键AD参数区
  自定义情境编辑器 / 网页端小智语音 / 设备健康 / 事件日志
```

### 14.2 状态颜色

| 状态 | 建议颜色 | 含义 |
| --- | --- | --- |
| 普通感知 | 蓝色 | 正在采集或参与当前情境 |
| 可能情境 | 紫色 | 系统根据证据给出的候选结果 |
| 关注或证据不足 | 黄色 | 需要用户确认、预热或缺少数据 |
| 普通执行动作 | 橙色 | 风扇、舵机、继电器等已执行 |
| 正常/已确认 | 绿色 | 数据新鲜且状态已确认 |
| 安全中断 | 红色 | MQ2、火焰等已确认风险 |
| 漏水辅助色 | 蓝红 | 水滴触发及低压保护 |
| 离线/未知 | 灰色 | 没有新鲜数据或传感器失效 |

安全红色的优先级高于普通模式颜色。风险解除后，卡片返回当前情境状态，而不是停留在旧报警颜色。

### 14.3 历史数据与配置

- 浏览器使用IndexedDB保存近期传感器趋势、情境变化、用户纠正和事件记录。
- `localStorage`只保存界面偏好、配置索引和小体量兜底信息，不堆积大量遥测。
- 配置修改后先显示“等待开发板确认”，收到 `ack` 后再显示“已生效”。
- 历史图表必须标注数据断点，不用连线掩盖离线区间。
- 没有实时遥测时，旧历史仍可查看，但当前结论必须显示未知。

## 15. 网页端小智语音入口

语音入口沿用相邻“初中”项目的网页链路：用户可以自然地表达需求，不要求逐字说出固定口令；网页语音服务把自然表达归并为有限、安全、可测试的意图。建议覆盖以下表达方向：

- “现在最可能是什么情境？”
- “为什么判断为学习情境？”
- “有哪些反向证据？”
- “切换到安静休息。”
- “进入湿热通风。”
- “确认当前判断。”
- “这次判断不正确。”
- “把声音阈值调高一点。”
- “当前有没有安全异常？”
- “蜂鸣器静音。”

语音实现边界：

- 浏览器麦克风录音先提交到 `/api/voice/transcribe`，识别文字再连同当前情境摘要提交到 `/api/voice/intent`；文本测试可直接从第二步开始。
- `/api/voice/intent`可使用已配置的大模型解析自然表达，并在服务不可用时回退到本地语义规则；API密钥只保存在服务器运行环境，不进入浏览器或Git。
- 输入句式可以自然变化，但输出只允许 `queryContext`、`explainContext`、`setMode`、`confirmContext`、`correctContext`、`setThreshold`、`querySafety`、`muteBuzzer`或`unknown`。
- 所有控制意图必须再转换为固定白名单 `command`，并等待开发板返回同ID的 `ack`；低置信度、参数越界和未知意图不得控制设备。
- 自由问答不能直接生成风扇、舵机、继电器或安全设备动作。
- GPIO4声音模块只输出强度，不采集或保存原始音频。
- 浏览器麦克风是可选入口，必须由用户主动授权。
- 如使用服务器STT，音频只在用户发起识别时发送到已配置服务；项目默认不长期保存原始音频，并应在界面说明实际数据路径。
- “网页显示识别成功”不等于设备控制成功；PIO主路线必须检查服务端意图、浏览器白名单、`command`、`ack`和真实执行器状态。只有将来切回小智统一固件路线时才检查板内MCP工具注册。

## 16. 隐私与能力边界

本作品可以表述为“规则驱动的多传感情境匹配系统”，不宣称：

- 识别具体人员、身份、年龄或健康状况。
- 判断房屋内准确人数。
- 通过声音强度识别说话内容、情绪或求救语义。
- 使用普通PIR判断所有人员是否已经离开。
- 使用DHT、MQ2或普通火焰模块作专业诊断。
- 在未经训练和验证的情况下宣称AI识别准确率。
- 没有电能计量模块却声称精确节电量、节省费用或碳减排量。

Dashboard使用“情境匹配度”，明确表示规则条件的满足程度，不使用“机器学习置信度”误导评委。

## 17. 五分钟现场演示流程

### 第一步：30秒说明问题

“许多智能家居依赖摄像头，或者看到一个数值就直接控制设备。我们的作品不拍摄家庭成员，而是组合多种不涉及身份的环境线索，并把每次判断的依据展示出来。”

### 第二步：30秒证明真实数据

- 展示开发板、USB/WebSocket、数据帧和最近遥测时间。
- 遮挡光敏、触发PIR或改变声音强度，证明网页数据来自实物。
- 指出声音模块只显示强度，不录音。

### 第三步：50秒展示情境推理

- 进入专注学习。
- 展示PIR活动、光照和声音如何形成支持证据或反向证据。
- 指出覆盖率、匹配度和缺失证据。
- 展示低压学习灯、RGB提醒或允许的通风动作。

### 第四步：40秒用户纠正

- 使用8键AD、Dashboard或小智确认或否定当前判断。
- 调整一个阈值或权重。
- 指出命令编号和真实 `ack`，证明配置进入开发板而不是只改网页文字。

### 第五步：50秒场景联动

- 把DHT阈值调整到适合现场演示的安全范围。
- 进入湿热通风，展示风扇、舵机和RGB联动。
- 说明动作来自哪些有效证据，以及DHT失效时系统会如何退出判断。

### 第六步：50秒安全中断

- 使用少量清水安全触发水滴模块。
- 证明普通情境立即让位于安全中断，低压继电器进入保护状态。
- MQ2和火焰只通过安全测试或mock补充展示，不使用危险气体、烟雾或明火。

### 第七步：40秒历史与自定义

- 展示近期情境变化、用户纠正和动作时间轴。
- 展示一个由用户保存的自定义模板。
- 返回实时侦测，确认安全中断经过持续恢复和人工确认后才解除。

## 18. 验收标准

### 18.1 方案与接线

- [ ] 所有传感器和执行器与本文件GPIO表一致。
- [ ] RFID保持禁用，不占用 `GPIO11/12/13`。
- [ ] MQ2的5V AO已分压到 `0~3.3V`。
- [ ] 继电器只控制低压负载。
- [ ] 外部电源执行器与N16R8共地。
- [ ] 上电默认状态经过逐项真机检查。

### 18.2 固件与协议

- [ ] 固件构建通过。
- [ ] 串口能看到真实 `hello` 和持续 `telemetry`。
- [ ] 所有模式、配置和纠正命令都返回匹配命令编号的 `ack`。
- [ ] 情境评分正确处理证据缺失、覆盖不足、分数接近和结果未知。
- [ ] DHT使用独立低频读取并能识别持续失效。
- [ ] MQ2预热期不触发危险动作。
- [ ] 火焰优先级可以覆盖MQ2排风，火焰确认后风扇停止。
- [ ] 自定义规则不能覆盖安全中断。
- [ ] 数据中断进入未知状态，不继续显示旧结论。

### 18.3 Dashboard与隐私

- [ ] 首屏显示开发板、USB、WebSocket、MQTT和数据新鲜度。
- [ ] 支持证据、反向证据和缺失证据均清楚可见。
- [ ] 实时趋势、历史情境和用户纠正能够保存和查看。
- [ ] 没有实时数据时显示“等待实时数据”。
- [ ] 网页不采集图像或视频。
- [ ] GPIO4声音模块只显示强度数值。
- [ ] 可选麦克风入口明确显示授权与STT数据路径。
- [ ] 语音指令经过白名单转换，不能自由生成危险动作。
- [ ] 拔掉开发板后，网页明确进入离线状态。

### 18.4 模拟与真板

- [ ] mock模式能够演示六种情境、安全中断、用户纠正和 `ack`。
- [ ] 至少完成一次真板光敏、声音、PIR和水滴变化验证。
- [ ] 风扇、舵机、低压继电器、RGB和蜂鸣器分别完成低压真机测试。
- [ ] Dashboard显示的动作与真实执行器一致。
- [ ] 手动控制无法覆盖已确认的安全联动。
- [ ] 至少完成一次“用户纠正—配置保存—刷新恢复—板端确认”闭环。

## 19. 实施顺序

本设计获得最终文档复核后，建议按以下顺序实施：

1. 冻结GPIO、项目画像、模式名、协议字段和安全默认值。
2. 先实现纯规则情境评分模块及自动化测试。
3. 扩展mock网关，覆盖六种情境、证据缺失、安全中断和 `ack`。
4. 制作Dashboard最小闭环，先显示真实/模拟数据、证据和动作原因。
5. 实现固件采集、板端安全状态机、情境评分和命令处理。
6. 接入8键AD、自定义配置和网页端自然语言意图链路。
7. 完成真板低压执行器验证。
8. 最后处理公网HTTPS/Web Serial、WSS/MQTT和移动端展示。

不得先做静态漂亮页面，再用默认值冒充真实情境判断。

## 20. 真机烧录边界

当前设计阶段不执行烧录。后续进入真机阶段时：

1. 先完成mock、协议测试和固件构建。
2. 使用短而稳定的USB线，烧录前断开风扇、舵机、RGB和继电器等高电流负载。
3. 停止占用串口的网关、监视器和其他进程。
4. 保留小智Wi-Fi、NVS和激活数据；未经明确授权不执行全片擦除或全量覆盖。
5. 先核对当前固件的分区表、应用偏移和生成文件。当前小智统一固件app地址为 `0x400000`；本项目PIO应用地址为 `0x10000`。两条路线不得混用，技能旧稿中的 `0x100000` 不适用于当前已核实布局。
6. 写入后必须针对实际写入地址和实际二进制运行 `verify_flash`；只有出现摘要匹配才能记录为烧录成功。
7. PIO主路线启动后检查主板身份、项目画像、遥测GPIO、网页语音服务、双重白名单、命令 `ack`和真实执行器动作；只有小智统一固件路线才检查板内MCP工具注册。
8. 如果macOS串口消失或出现 `Device not configured`，先检查USB供电、线材、端口变化和串口占用，不先改固件逻辑。

## 21. 当前不实施的范围

- RFID门禁。
- `220V`电器控制。
- 摄像头、人脸识别、人数统计或身份识别。
- 对声音内容、情绪或求救语义的识别。
- 未经验证的机器学习训练或准确率宣传。
- 真实消防、燃气阀、水阀、医疗或人员看护控制。
- 精确电能、费用或碳排放计量。
- 未经用户确认自动重写情境规则。
- 让自由文本模型直接控制危险执行器。

## 23. 一句话答辩总结

> 我们没有用摄像头观察家庭成员，也没有看到一个数值就直接下结论；这座房屋会组合多种不涉及身份的环境线索，告诉用户它为什么这样判断、还缺少什么证据，并允许用户纠正和保存自己的生活规则。
