# N16R8 无摄像头家庭情境侦探屋自然语言控制扩展设计

> 日期：2026-07-16  
> 状态：方案A及两部分设计已经用户确认，本文是实施前的书面设计规格。  
> 项目：`smartlife-junior-context`  
> 项目画像：`smartlife-junior-context-detective-v1`  
> 公网入口：`https://zhihuishenghuochu2.ilelezhan.cn/`

## 1. 决策摘要

本批次把现有网页语音从“情境查询、模式切换、判断纠正和阈值调整”扩展为完整、可审计的自然语言控制入口：

1. 支持六种模式切换。
2. 支持风扇、舵机窗、低压LED继电器、蜂鸣器和RGB灯环五类执行器直接控制。
3. 支持光照、声音、温度、湿度、人体、8键AD、MQ2、水滴和火焰九项读数查询。DHT是一个模块，但温度和湿度作为两个独立测量项回答。
4. 支持光照、声音、温度、湿度和MQ2五类阈值调整。
5. MQ2、水滴、火焰等安全采样与安全状态不能被语音关闭、旁路或伪造为正常。

采用一个通用的`setActuator`语音意图，而不是为五种执行器分别增加五套意图。服务端只产生结构化、单操作的意图，Dashboard再次清洗后复用固件已经存在的`command.actuator`合同。这样可以保持服务端、浏览器、Mock和固件只有一份执行器参数表。

本批次不改变固定GPIO，不给自由文本提供GPIO、PWM、任意舵机角度或多执行器JSON出口，也不在主板上运行自然语义模型。

## 2. 用户可见能力

### 2.1 六种模式

| 中文名称 | 协议值 | 示例表达 |
| --- | --- | --- |
| 实时侦测 | `detect` | “返回侦测”“看看家里现在什么情况” |
| 专注学习 | `study` | “进入学习模式”“我要专心写作业” |
| 安静休息 | `rest` | “进入休息模式”“我要安静休息” |
| 湿热通风 | `ventilation` | “进入通风模式”“启用湿热通风情境” |
| 无人节能 | `energy` | “进入节能模式”“家里没人了” |
| 自定义情境 | `custom` | “进入自定义模式”“使用我的情境” |

模式表达生成`setMode`。模式切换与执行器直接控制是两种不同意图：

- “我热了”按用户确认的口径生成风扇70%的直接控制，保持当前模式不变。
- “进入湿热通风模式”切换`ventilation`，由板端情境规划器决定联动目标。
- 一次语音只生成一个标准操作，不能同时切模式并暗中发送多个设备命令。
- 现有手动覆盖不会因模式切换被静默删除；页面必须持续显示哪些执行器仍处于人工覆盖。用户可以对单个执行器说“恢复自动”。

### 2.2 五类执行器直接控制

服务端意图统一为：

```json
{
  "intent": "setActuator",
  "actuator": {"fan": 70},
  "confidence": 0.92
}
```

`actuator`必须且只能包含一个字段。

| 执行器 | 语音允许值 | 示例表达 | 标准载荷 |
| --- | --- | --- | --- |
| 风扇 | 整数`0~100`或`auto` | “我热了”“风扇开到50%”“最大风”“关闭风扇”“风扇恢复自动” | `{"fan":70}`、`{"fan":50}`、`{"fan":100}`、`{"fan":0}`、`{"fan":"auto"}` |
| 舵机窗 | `ventilation-open`、`safety-closed`或`auto` | “打开窗户”“关闭窗户”“窗户恢复自动” | `{"servo":"ventilation-open"}`、`{"servo":"safety-closed"}`、`{"servo":"auto"}` |
| 低压LED继电器 | 布尔值或`auto` | “开灯”“关灯”“灯恢复自动” | `{"relay":true}`、`{"relay":false}`、`{"relay":"auto"}` |
| 蜂鸣器 | 布尔值或`auto` | “蜂鸣器响一下”“停止蜂鸣器测试”“蜂鸣器恢复自动” | `{"buzzer":true}`、`{"buzzer":false}`、`{"buzzer":"auto"}` |
| RGB灯环 | 白名单颜色、`off`或`auto` | “灯环变蓝”“灯环变紫”“关闭灯环”“灯环恢复自动” | `{"rgb":"blue"}`、`{"rgb":"purple"}`、`{"rgb":"off"}`、`{"rgb":"auto"}` |

风扇自然语义采用确定性基线：

- “我热了”“有点热”“吹点风”“打开风扇”映射为70%。
- “小风”“低速风”映射为30%。
- “太热了”“最大风”“强力通风”映射为100%。
- “关闭风扇”“不要吹了”映射为0%。
- “风扇开到N%”接受`0~100`整数。
- “风大一点”“风小一点”只有在当前真板或显式Mock的风扇实际值新鲜时，才按20个百分点增减并限制在`0~100`；没有新鲜基准时不猜值，提示用户说出明确百分比。

RGB网页语音首批只开放`orange/cyan/yellow/red/green/blue/purple/off/auto`。`blue-red`保留给安全联动，不能由普通语音伪造；`study`和`blue-low`继续由情境规划器使用。GPIO46尚无人工可见亮灯证据，因此即使收到成功ACK，页面仍只能写“主板已确认输出状态，实物待验收”。

蜂鸣器`true`复用固件现有短脉冲测试，不表示持续报警。`false`停止测试并建立人工关闭覆盖；`auto`释放该覆盖。“蜂鸣器静音”继续使用独立的`muteBuzzer`，只把`buzzerEnabled`设为`false`，不会清除MQ2、水滴、火焰风险或其他保护动作。

### 2.3 九项传感器查询

新增只读意图：

```json
{
  "intent": "querySensor",
  "sensor": "temperature",
  "confidence": 0.94
}
```

允许的`sensor`只有：

```text
light, sound, temperature, humidity, pir,
keypad, mq2, water, flame
```

| 查询示例 | 回答依据 |
| --- | --- |
| “现在温度多少” | `sensors.temperature`、`sensorValid.temperature`和`sensorAgeMs.temperature` |
| “湿度怎么样” | `sensors.humidity`及其有效性和数据年龄 |
| “家里有人吗” | PIR当前布尔值；只表达指定通道近期是否检测到活动，不推断身份或人数 |
| “MQ2数值多少” | MQ2原始ADC和是否仍在预热；不能把ADC换算成未经标定的燃气浓度 |
| “漏水了吗” | 水滴传感器当前输入与安全状态；不能把未触发描述为专业漏水系统证明安全 |
| “检测到火焰了吗” | 火焰输入与安全状态；不能判断真实火势等级 |

查询在Dashboard本地读取当前数据源，不发送板端命令，也不要求`hello.capabilities.commands`新增查询能力。必须同时满足：

1. 当前帧项目和画像匹配。
2. 遥测总新鲜度不超过3500ms。
3. 对应`sensorValid`为`true`。
4. 数值不是`null`，对应数据年龄仍在页面允许范围内。

条件不满足时回答“当前没有有效的实时数据”，不使用默认值、旧值或其他项目数据补齐。Mock值必须明确标为模拟数据。

“关闭MQ2传感器”“不要检测漏水”“忽略火焰报警”等句子不是查询，必须返回安全拒绝，不能落入`querySensor`或任何控制意图。

### 2.4 五类阈值调整

继续使用现有`setThreshold`合同，每次只能修改一个字段：

| 字段 | 范围 | 步长 |
| --- | --- | --- |
| `lightThreshold` | `0~4095` | `100` |
| `soundThreshold` | `0~4095` | `50` |
| `temperatureThreshold` | `10~45` | `1` |
| `humidityThreshold` | `20~95` | `5` |
| `mq2Threshold` | `0~2600` | `50` |

MQ2语音阈值不能高于暂定安全基线2600，防止用极高阈值变相关闭报警。阈值只保存在RAM，复位后恢复固件默认值，页面必须说明这一点。相对表达“调高一点/调低一点”只有在当前新鲜遥测包含对应阈值时才计算；否则要求用户说出明确值。

当前真板仍运行`0.4.0`，仓库候选为`0.4.1`。Dashboard只有从当前数据源的匹配`hello.capabilities.commands`看到`setThreshold`时才允许发送；未声明时显示“固件待升级”，不把语义解析成功写成阈值已生效。

## 3. 语音意图合同

扩展后的服务端意图白名单为：

```text
queryContext
explainContext
querySafety
querySensor
setMode
setActuator
confirmContext
correctContext
setThreshold
muteBuzzer
unknown
```

分类如下：

| 类别 | 意图 | 是否发送主板命令 |
| --- | --- | --- |
| 只读 | `queryContext`、`explainContext`、`querySafety`、`querySensor` | 否 |
| 控制 | `setMode`、`setActuator`、`confirmContext`、`correctContext`、`setThreshold`、`muteBuzzer` | 是，但必须通过能力与链路检查 |
| 拒绝 | `unknown` | 否 |

控制意图最低置信度保持0.6。模型结果即使置信度较高，只要出现多意图、多执行器、未知字段、任意GPIO、PWM、角度、越界数值或安全旁路，服务端仍改写为`unknown`；Dashboard再执行一次相同边界的校验。

意图优先级必须避免旧规则把设备表达误判成模式：

1. 安全旁路拒绝。
2. 明确传感器查询。
3. 明确执行器直接控制。
4. 阈值调整。
5. 情境确认、纠正和模式切换。
6. 情境/安全解释查询。
7. `unknown`。

因此“我热了”稳定落入`setActuator fan=70`；只有明确出现“模式、情境、进入、切换”等模式词或完整模式名时，才落入`setMode`。

## 4. 端到端链路

```text
首页麦克风或文本降级
  -> POST /api/voice/transcribe（麦克风路径）
  -> POST /api/voice/intent + 安全裁剪后的新鲜上下文
  -> 讯飞星火结构化语义
  -> 服务端规则和白名单清洗
  -> dashboard/voice-core.js第二次清洗
  -> 只读查询在Dashboard回答
     或
     能力检查后生成带唯一ID的标准command
  -> Web Serial或项目独立WSS链路
  -> N16R8/显式Mock返回同ID ack
  -> 后续telemetry观察真实逻辑状态
```

发送给语义服务的上下文只保留完成本次判断需要的字段：模式、候选情境、告警、九项传感器、传感器有效性与年龄、五项阈值、执行器目标、执行器实际值和安全覆盖状态。不得包含供应商密钥、MQTT凭据、Wi-Fi、完整日志或无关设备信息。

### 4.1 标准命令示例

“我热了”最终只能转换为：

```json
{"type":"command","project":"smartlife-junior-context","id":"voice-...","actuator":{"fan":70}}
```

“打开窗户”最终只能转换为：

```json
{"type":"command","project":"smartlife-junior-context","id":"voice-...","actuator":{"servo":"ventilation-open"}}
```

“把温度阈值设为28度”最终只能转换为：

```json
{"type":"command","project":"smartlife-junior-context","id":"voice-...","set":{"temperatureThreshold":28}}
```

每条命令只含一个操作。所有成功与失败必须返回相同ID的`ack`；ACK成功后仍需观察后续遥测，不能直接把计划值复制为真实状态。

## 5. 首页与语音工作台反馈

首页语音卡保持紧凑，但最近一次闭环必须分层显示：

1. **识别原话**：例如“我热了”。
2. **语义结果**：例如“直接控制风扇，目标70%”。
3. **发送状态**：未发送、等待链路、等待同ID ACK、ACK拒绝或ACK成功。
4. **真实观察**：最新遥测中的实际值、安全覆盖状态和实物验收标签。

页面不得把以下概念合并：

- 识别成功不等于意图合法。
- 意图合法不等于命令已经发送。
- ACK成功不等于后续遥测已经观察到。
- 遥测显示GPIO输出状态不等于风扇、舵机、继电器或RGB已经完成人工实物验收。
- WSS/MQTT在线不等于真板在线。

首页增加有代表性的快捷表达，但不堆满全部口令：

```text
我热了
打开窗户
开灯
灯环变蓝
蜂鸣器响一下
现在温度多少
```

`#voice`工作台提供完整能力说明、九项查询入口、五类阈值示例、人工覆盖状态和安全拒绝说明。首页与工作台继续共用同一个语音会话状态机和同一份白名单。

## 6. 安全优先级

安全优先级继续由板端状态机决定，网页语音不得复制或替代安全判断：

```text
火焰 > MQ2 > 水滴 > 传感器故障 > 普通模式/网页语音/手动控制
```

固定规则：

1. 已确认风险可以覆盖同一执行器的语音人工目标。
2. MQ2或火焰风险下，语音不能把受安全控制的风扇、舵机、继电器、蜂鸣器或RGB强制恢复普通状态。
3. “关闭MQ2/水滴/火焰检测”“忽略报警”“一直认为安全”等请求直接拒绝。
4. 语音不能修改`sensorValid`、`sensorAgeMs`、`alerts`、`safety`或硬件验收字段。
5. `muteBuzzer`只静音声音；风险事件、继电器保护、RGB提示和页面告警继续保留。
6. 不使用真实燃气、危险烟雾或明火验证语音功能。

如果语音目标被安全状态覆盖，页面要同时显示“语音计划值”和“安全覆盖后的真实目标”，并明确给出风险原因。

## 7. 能力兼容与实施边界

| 能力 | 处理方式 |
| --- | --- |
| 六种模式 | 复用`setMode`，发送前要求匹配`hello`声明`setMode` |
| 五类执行器 | 复用已有`setActuator`，发送前要求匹配`hello`声明`setActuator` |
| 九项查询 | Dashboard只读新鲜遥测，不需要新增固件命令 |
| 五项阈值 | 复用`setThreshold`；当前真板未声明时显示固件待升级 |
| 安全传感器保护 | 复用板端安全引擎，网页只增加拒绝测试和诚实反馈 |

本批软件实现预计不需要扩展固件串口结构；仍要运行固件契约测试和PIO纯编译，证明前端/服务器修改没有破坏现有协议。没有新的烧录授权，本批次禁止`upload`、`write_flash`、`erase_flash`、串口执行器命令或真板动作验收。

公网部署只更新本项目的`/home/ubuntu/smartlife-context-detective`、`smartlife-context-*`服务和独立站点资源。部署前后必须核对其他SmartLife服务状态/PID及原站点HTTP状态；不得改动、重启或覆盖相邻项目。

## 8. 错误处理

| 情况 | 页面处理 |
| --- | --- |
| STT失败或无文字 | 显示原因，保留文本降级入口，不发送命令 |
| 模型不可用 | 只对确定性规则明确命中的表达降级；否则`unknown` |
| 含糊表达或对象不明 | 提示用户换一种说法，不默认模式或设备 |
| 多操作表达 | 拒绝并提示一次只控制一个设备或一种模式 |
| 数值或枚举越界 | 双重白名单拒绝，不截断成看似成功的值 |
| 没有新鲜遥测 | 查询回答数据不可用；相对调整不猜基准值 |
| 当前`hello`缺少能力 | 显示“固件待升级/当前设备不支持”，不发送 |
| 实时链路不可用 | 保留计划动作，显示“未发送到主板” |
| 同ID ACK超时或拒绝 | 显示“主板未确认”及非敏感错误码 |
| ACK成功但遥测未变化 | 分开显示ACK与观察结果，不声称设备已动作 |
| 安全覆盖 | 显示语音目标、覆盖目标和风险原因 |

## 9. 测试策略

实施仍遵守“先写失败测试，再做最小实现”。

### 9.1 Python语音服务测试

- 六种模式的明确自然表达。
- 五种执行器的开启、关闭、自动、数值和枚举。
- 固定用例：“我热了”必须得到`setActuator`和`{"fan":70}`，不能得到`setMode`。
- 九项传感器查询及同义词。
- 五项阈值的明确值、相对值、步长、边界和越界拒绝。
- MQ2阈值高于2600必须拒绝。
- 模型返回多执行器、GPIO、PWM、任意角度、未知颜色或混合操作时必须变为`unknown`。
- “关闭烟雾检测”“忽略火焰报警”“不要监测漏水”等安全旁路必须拒绝。
- 星火不可用时，确定性规则只降级明确表达；含糊文本仍为`unknown`。

### 9.2 Dashboard合同测试

- `voice-core.js`与服务端使用相同意图、模式、执行器、传感器和阈值白名单。
- `setActuator`只能包含一个执行器，且转换为现有单操作`command.actuator`。
- `querySensor`不生成主板命令，只读取新鲜、有效遥测。
- 旧遥测、无效传感器和`null`值不能回答为正常数值。
- `setActuator`和`setThreshold`按匹配`hello.capabilities.commands`放行。
- 语音状态区分别展示文字、意图、计划、同ID ACK、后续遥测和实物验收标签。
- 安全覆盖时计划值与真实目标不合并。
- 390px宽度无横向溢出，键盘操作与减少动态效果状态可用。

### 9.3 Mock与浏览器闭环

显式Mock至少完成：

1. “我热了” -> 风扇70% -> 同ID ACK -> 新遥测。
2. “打开窗户” -> 通风位 -> ACK -> 新遥测。
3. “开灯” -> 继电器开启 -> ACK -> 新遥测。
4. “灯环变蓝” -> 蓝色 -> ACK -> 新遥测，并继续标记Mock/实物未验收。
5. “蜂鸣器响一下” -> 短脉冲合同。
6. 九项传感器逐项查询。
7. 五项阈值合法调整和非法调整。
8. 火焰、MQ2、水滴场景对语音人工目标的安全覆盖。
9. 安全传感器关闭语句被拒绝且没有命令进入账本。
