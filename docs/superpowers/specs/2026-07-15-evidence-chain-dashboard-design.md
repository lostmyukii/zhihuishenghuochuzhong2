## 1. 设计结论

Dashboard 采用“证据链总览 + 六大工作台”结构。

首页服务三分钟现场演示，信息顺序固定为：

```text
真实采集
  -> 情境推理
  -> 控制来源
  -> 联动结果
  -> 事件记录
```

页面不出现“评分”“得分”“高分”“功能1—功能5”等比赛字样，也不显示任何虚构分数。页面通过真实状态、完整操作和可追溯证据自然体现作品能力。

参考站的功能能力必须完整保留，但不得复制其“智能住宅管家”名称、项目身份、场景名、部署标识或视觉布局。最终页面必须始终是“N16R8 无摄像头家庭情境侦探屋”。

## 2. 设计目标

### 2.1 首要目标

1. 评委无需寻找功能，沿首页从左到右即可理解作品闭环。
2. 一眼区分真板、Mock、缓存、离线和未验收硬件。
3. 一眼区分传感数据、情境判断、计划动作和真实动作。
4. 支持本地按键、网页控制和网页语义三种独立控制来源。
5. 保留完整配置、调试、注册、语音和历史追溯工作台。
6. 桌面投屏和手机浏览均无需横向滚动即可完成核心演示。

### 2.2 非目标

1. 不把所有控件同时塞进第一屏。
2. 不把 Mock 页面包装成真板演示。
3. 不把声音原始值标成 `dB(A)`。
4. 不把 MQ2 原始 ADC 标成空气质量或 `ppm`。
5. 不在浏览器保存 API 密钥、MQTT 密钥、Wi-Fi 或服务令牌。
6. 不在本次页面重构中改变既有 GPIO、串口 JSON 合同或安全优先级。

## 3. 视觉系统

### 3.1 设计主题

主题为“现代家庭侦探档案室 + 工程蓝图”。它表达的是可解释推理和家庭环境证据，不使用警察、犯罪、监控镜头、放大镜堆叠等刻板侦探图形。

唯一主视觉是横向“环境证据链轨道”：五个节点连接真实采集、情境推理、控制来源、联动结果和事件记录。除这条轨道外，不增加第二套大型装饰系统。

### 3.2 色彩令牌

只定义六个核心颜色；浅色卡片通过透明度生成，不增加无意义主题色。

| 令牌 | 色值 | 用途 |
| --- | --- | --- |
| `ink` | `#10243E` | 主文字、导航、深色状态条 |
| `paper` | `#F4F7FA` | 页面背景、工程网格底色 |
| `evidence` | `#0A9A91` | 新鲜数据、证据链、当前判断 |
| `verified` | `#278866` | 已确认、有效、真实成功回执 |
| `pending` | `#D58B26` | 待校准、待实物验收、备用链路 |
| `alert` | `#C84C4C` | 真实告警、反向证据、危险覆盖 |

状态颜色不得只靠颜色表达，必须同时显示文字或图标。

### 3.3 字体角色

| 角色 | 字体栈 | 用途 |
| --- | --- | --- |
| 叙事标题 | `Songti SC, STSong, serif` | 项目名、情境结论、一级标题 |
| 操作正文 | `PingFang SC, Microsoft YaHei, sans-serif` | 导航、按钮、说明、表格 |
| 工程数据 | `SFMono-Regular, JetBrains Mono, monospace` | GPIO、命令ID、时间、数值 |

禁止使用通篇等宽字体，也不加载比赛现场可能不可用的远程字体。

### 3.4 卡片与背景

- 页面使用浅灰蓝纸张背景和极细工程网格。
- 卡片为白色、细描边、轻阴影，圆角控制在 `12—18px`。
- 不使用大面积玻璃拟态、霓虹、炫彩渐变或纯黑背景。
- 不用大量 KPI 数字卡代替解释；数字必须属于真实传感或推理字段。

## 4. 信息架构

顶部固定六个一级入口：

1. `侦探总览`
2. `设备注册`
3. `情境联动`
4. `网页语音`
5. `调试台`
6. `数据日志`

“连接中心”不是第七个顶级页面，而是总览状态条的展开面板，并在完整工作台底部提供快捷入口。

### 4.1 桌面布局

```text
┌──────────────────────────────────────────────────────────────┐
│ 品牌 / 项目身份                         六个一级入口           │
├──────────────────────────────────────────────────────────────┤
│ 开发板 | Web Serial | WebSocket | MQTT | 数据源 | 情境 | 年龄 │
├──────────────────────────────────────────────────────────────┤
│ ①真实采集 ── ②情境推理 ── ③控制来源 ── ④联动结果 ── ⑤记录 │
├─────────────┬──────────────────────┬──────────────────────────┤
│ 官方关键输入 │ 房屋线索地图          │ 8键 / 网页 / 网页语义      │
│ 额外安全输入 │ 当前判断              ├──────────────────────────┤
│ 有效性/年龄  │ 支持/反向/缺失证据     │ 计划动作 | 真实状态 | 验收  │
│ 单位口径     │ 覆盖率/匹配度/待确认   │ 最近事件与同ID ACK          │
├─────────────┴──────────────────────┴──────────────────────────┤
│ 设备注册 | 情境联动 | 网页语音 | 调试台 | 数据日志 | 连接中心 │
└──────────────────────────────────────────────────────────────┘
```

建议桌面断点：

- `>= 1440px`：三列总览，比例约 `1 : 2.15 : 2`。
- `1024—1439px`：采集与推理同排，控制/联动结果移到下一排。
- `768—1023px`：两列工作区，顶部导航允许水平滚动但不换成不可发现的汉堡菜单。

### 4.2 手机布局

手机端遵循同一逻辑，不按桌面视觉简单缩小：

```text
┌──────────────────────┐
│ 项目名       连接状态 │
│ 横向一级入口          │
├──────────────────────┤
│ 真板 / 数据源 / 年龄   │
├──────────────────────┤
│ 当前判断 + 告警        │
├──────────────────────┤
│ ①关键真实采集 2列      │
├──────────────────────┤
│ ②证据链               │
├──────────────────────┤
│ ③三种控制来源          │
├──────────────────────┤
│ ④计划 / 真实状态       │
├──────────────────────┤
│ ⑤最近三条事件          │
└──────────────────────┘
```

- 手机总览默认只展示官方关键输入、当前安全状态和最近三条事件。
- 额外传感器可展开，不让九张传感器卡压过情境结论。
- 执行器按钮最小触控高度 `44px`。
- `390px` 宽度下不得出现页面级横向滚动。

## 5. 总览页面

### 5.1 项目身份区

必须显示：

- `N16R8 无摄像头家庭情境侦探屋`
- `无摄像头 · 非录音 · 多源环境证据`
- 当前固件、项目和画像可放入身份详情弹层，不占首屏标题空间。

禁止显示参考项目的“智能住宅管家”“可迁移智能中枢”等名称。

### 5.2 实时状态条

状态条必须包含：

| 项目 | 可能值 |
| --- | --- |
| 开发板 | 真板在线 / 真板遥测在线 / 模拟板在线 / 等待实时数据 / 离线 |
| Web Serial | 已连接 / 未连接 / 浏览器不支持 / 授权失败 |
| WebSocket | 已连接 / 备用 / 重连中 / 离线 |
| MQTT | 后续接入 / 已连接 / 离线；没有真实连接时不得显示在线 |
| 数据源 | 真板 / Mock / 缓存 / 无数据 |
| 当前情境 | 六种合法模式之一 |
| 最后遥测 | 相对年龄，例如 `0.2秒前` |

点击状态条打开“连接中心”，显示 Web Serial 主链路、Python WebSocket 备用链路、未来 WSS/MQTT 信息和当前数据来源说明。

#### 5.2.1 未捕获启动身份的真板遥测

开发板只在启动时发送一次`hello`，浏览器可能在主板已经运行后才打开Web Serial，从而持续收到新鲜`telemetry`却没有捕获启动身份。此时不得显示“无实时数据”，也不得直接升级为完整“真板在线”。

新增独立状态`real-telemetry`，必须同时满足：

1. `telemetry.type=telemetry`。
2. `project=smartlife-junior-context`且`profileId=smartlife-junior-context-detective-v1`。
3. `mock`不为`true`。
4. 遥测年龄不超过`3500ms`。
5. 来源明确为Web Serial或真实Python网关。
6. 当前同源`hello`缺失；如果收到身份冲突的`hello`，不得进入该状态。

该状态的开发板文案固定为“真板遥测在线”，数据源文案固定说明“未捕获启动身份”，使用待确认色。后续收到同源且匹配的`hello`后升级为`real-live`；遥测过期后仍进入`stale`并清空当前结论。Mock、外项目、缺少画像或来源不明的帧不得进入该状态。

### 5.3 真实采集

第一层置顶四类关键输入：

1. 温度与湿度。
2. 声音相对强度。
3. 人体感应。
4. 空气/气体类输入；在当前 MQ2 未获得正式空气质量依据前，只显示 `MQ2 原始ADC` 和预热状态。

第二层展示光照、8键AD、水滴、火焰等额外输入。

每个输入统一显示：

- 名称。
- GPIO。
- 当前数值或布尔状态。
- 诚实单位。
- `sensorValid`。
- `sensorAgeMs`。
- 预热、去抖或故障原因。

旧值超过新鲜度阈值后必须变为“未知/已过期”，不能继续伪装实时。

### 5.4 情境推理

主区域使用六房间线索地图，不要求复刻真实沙盘照片。房间与模块映射固定为当前项目合同：

| 房间 | 主要线索或设备 |
| --- | --- |
| 书房 | 光敏、声音、低压继电器LED |
| 客厅 | 温湿度、8键AD、风扇 |
| 厨房 | MQ2、火焰 |
| 卫生间 | 水滴 |
| 卧室 | 舵机窗、RGB灯环 |
| 门厅 | PIR人体活动 |

中央判断显示：

- 当前候选情境。
- 覆盖率。
- 匹配度。
- `status`。
- 支持证据。
- 反向证据。
- 缺失证据。

当状态为 `uncertain` 或缺失关键输入时，必须使用“待确认/证据不足”，不能显示确定结论。

### 5.5 控制来源

总览同时显示三种独立来源：

1. `8键AD`
2. `网页控制`
3. `网页语义`

每张来源卡显示最近一次：

- 用户动作或原始输入。
- 标准化意图。
- 命令 `id`。
- 发送时间。
- 同 ID `ack`。
- 成功、失败或超时原因。

三种来源共用同一标准命令合同，不各自维护互不兼容的逻辑。

### 5.6 联动结果

五类执行器固定显示：

- 风扇 `GPIO11`
- 舵机 `GPIO9`
- 继电器LED `GPIO12`
- 蜂鸣器 `GPIO13`
- RGB灯环 `GPIO46`

每项必须同时展示：

```text
计划动作 | 板端实际状态 | 自动/手动来源 | 安全覆盖 | 实物验收状态
```

呈现规则：

- `actuatorTargets` 只能显示为计划动作。
- `actuators` 非空且来源真实时，才显示为板端实际状态。
- Mock 显示“模拟执行”，不得显示“真实动作”。
- 没有人工实物证据的设备显示“待实物验收”。
- GPIO13蜂鸣器可显示“已单项验证”，但不能据此推断其他执行器已验证。
- `hardwareVerified=false` 和 `calibrationRequired=true` 必须保持可见且可解释。

### 5.7 事件记录

总览仅展示最近 3—5 条关键事件，每条包括：

- 时间。
- 数据源。
- 情境或告警判断。
- 命令ID。
- `ack`。
- 计划动作。
- 板端实际状态。
- 实物验收标签。

点击进入完整数据日志。

## 6. 六大工作台

### 6.1 设备注册

完整保留参考站能力并使用当前13模块合同：

1. 主板身份识别。
2. GPIO端口扫描结果。
3. 13个模块选择。
4. 房间归类。
5. 情境绑定。
6. 本地保存。
7. 云端同步占位；没有真实服务时显示“未配置/后续接入”。

注册表固定包含以下13个模块：

| 模块 | GPIO | 类型 |
| --- | --- | --- |
| 光敏传感器 | `GPIO1` | 输入 |
| 声音传感器 | `GPIO4` | 输入 |
| 温湿度DHT | `GPIO14` | 输入 |
| PIR人体红外 | `GPIO5` | 输入 |
| 8键AD键盘 | `GPIO10` | 输入 |
| MQ2烟雾/燃气 | `GPIO2` | 输入 |
| 水滴传感器 | `GPIO8` | 输入 |
| 火焰传感器 | `GPIO45` | 输入 |
| 有源蜂鸣器 | `GPIO13` | 执行器 |
| 风扇 | `GPIO11` | 执行器 |
| 舵机 | `GPIO9` | 执行器 |
| 低压LED继电器 | `GPIO12` | 执行器 |
| RGB灯环 | `GPIO46` | 执行器 |

主题外观功能保留，但改为本项目主题：

- 档案纸（默认）。
- 工程蓝。
- 安全警戒。
- 节能清新。
- 夜间低光。

换主题只改变视觉令牌，不改变状态语义和安全颜色。

### 6.2 情境联动

只允许六个板端模式：

```text
detect, study, rest, ventilation, energy, custom
```

工作台内容：

- 六种情境卡。
- 当前规则解释。
- 输入条件与有效性。
- 计划动作预览。
- 安全覆盖原因。
- 阈值调整。
- 修改前后值。
- 同ID `ack`。
- 生效时间和恢复状态。

阈值调整必须执行范围校验。网页不能允许用户修改固定GPIO、安全优先级或任意PWM/GPIO表达式。

### 6.3 网页语音

完整保留：

- 麦克风设备列表。
- 刷新设备。
- 浏览器授权。
- 麦克风自检。
- 语音输入。
- 快捷语句。
- 文本测试。
- 原始文本、解析意图、标准命令、命令ID、`ack`和失败原因。

语义由浏览器或同域服务处理。主板继续保持：

```text
webVoiceIntent=true
localVoiceNlu=false
mcp=false
```

服务不可用时显示规则降级，不伪造识别成功；手动网页、8键AD和板端安全引擎继续可用。

### 6.4 调试台

完整保留五类执行器快捷控制。正式设计增加三道可见保护：

1. 当前数据源必须是真板，Mock只能模拟执行。
2. 首次真实动作要求用户单项确认。
3. 操作后等待同ID `ack`，再等待新的实际遥测。

调试台不提供“一键全部动作”。安全报警覆盖普通手动命令。

未获得新的明确硬件授权前，页面实现和测试只能使用 Mock，不得借页面重构自动触发真板执行器。

### 6.5 数据日志

完整保留并增强：

- 实时事件时间轴。
- 传感趋势。
- 按来源、模式、设备、告警、成功/失败过滤。
- 命令ID与 `ack` 对照。
- 计划动作与实际状态对照。
- 真板、Mock、缓存来源过滤。
- 导出 JSON 或 CSV。

日志必须限制内存数量并对导出内容做结构化清洗，不包含凭据。

### 6.6 侦探总览

总览不是其它工作台的重复堆叠，只显示完成三分钟演示所需的摘要。详细设置、批量表格和长历史全部进入对应工作台。

## 7. 告警设计

告警优先于普通情境，但不能永久遮挡全部页面。

告警条必须说明：

- 告警名称。
- 输入模块与GPIO。
- 当前值/状态。
- 阈值或确认规则。
- 板端采取的安全动作。
- 哪些实际动作已确认。
- 恢复条件。

告警颜色仅在真实危险状态使用红色。MQ2预热、传感器失效、硬件待验收使用琥珀色，不使用红色假装危险。

## 8. 交互与动效

- 状态改变使用 `120—180ms` 的颜色与位移过渡。
- 新证据进入时只高亮对应证据链节点，不让整页闪烁。
- 告警允许一次短促强调，随后稳定显示。
- 发送命令后按钮进入“等待ACK”；成功、失败、超时均恢复可操作状态。
- 支持 `prefers-reduced-motion`，关闭非必要位移和脉冲动画。
- 不使用自动轮播和无法暂停的持续动画。

## 9. 可访问性

- 所有按钮可键盘操作并有明确焦点环。
- 状态条和告警使用合适的 `aria-live`，避免每200ms遥测都朗读。
- 图表与颜色状态均提供文本替代。
- 表单使用真实 `label`。
- 对比度满足普通文本至少 `4.5:1`。
- 手机端按钮最小触控面积 `44×44px`。

## 10. 数据与安全边界

### 10.1 保持现有合同

- 串口每行一个完整JSON对象。
- 项目与画像必须匹配后才接受数据。
- 命令必须带 `id`。
- 未知类型、未知模式或无ID命令不得静默成功。
- Web Serial 和 Python 网关不得同时占用 CH340。

### 10.2 页面状态机

页面至少区分：

```text
waiting
real-live
mock-live
stale
offline
command-pending
command-ok
command-error
hardware-pending-verification
```

状态转换必须由新鲜 `hello / telemetry / ack` 驱动，不能仅由 WebSocket 打开或页面加载完成驱动。

### 10.3 凭据

STT、MQTT、WSS或公网部署所需密钥只在服务端环境变量保存。浏览器代码和Git不得出现明文密钥。
